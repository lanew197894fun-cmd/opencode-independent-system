## 啟動條件

- 當計散機系統閒置超過 3 分鐘時自動觸發
- 當用戶說「JI_DEBUG」或「計散機除錯」時手動觸發

## 功能

### 1. 計散機系統健康檢查

- 知識庫狀態
- 搜尋引擎索引
- 問題解決器模式
- 記憶系統
- 監控系統
- 安全防護

### 2. 核心模組檢測

- 模型管理器狀態
- 離線模式檢查
- 知識庫完整性
- 搜尋索引更新

### 3. 效能檢測

- 回應時間
- 記憶體使用
- CPU 使用率

### 4. 連線檢測

- VPN 連線狀態
- 24靈防護狀態
- RDP 攔截狀態
- 審計日誌

### 5. 自動修復

- 知識庫重新整理
- 搜尋索引重建
- 記憶體釋放
- 連線逾時處理

### 6. 報告生成

- 生成診斷報告
- 記錄問題到日誌
- 顯示修復建議

## 使用方式

```
# 手動觸發
「JI_DEBUG」
「計算機除錯」
「執行 Debug」

# 自動觸發
計散機閒置 3 分鐘後自動執行
```

## 閒置檢測機制

1. 記錄最後一次用戶互動時間
2. 定時檢查是否超過閒置閾值 (3 分鐘)
3. 觸發後執行完整診斷流程
4. 發現問題時主動嘗試修復
5. 修復完成後報告結果

## 快取優化（減少 API 消耗）

### 快取策略

| 檢查項目   | 快取 TTL | 說明               |
| ---------- | -------- | ------------------ |
| 知識庫狀態 | 300s     | 5分鐘內不重複檢查  |
| 搜尋索引   | 600s     | 10分鐘內不重複檢查 |
| 效能檢測   | 60s      | 1分鐘內不重複      |
| 連線狀態   | 120s     | 2分鐘內不重複      |

### 快取鍵命名

```
ji:health:{check_type}
ji:status:{module}
ji:performance:{metric}
```

### 檢查前先查快取

```
執行 Debug 前：
→ tool_cache_get(key: "ji:health:knowledge")
→ 有結果且未過期：跳過檢查
→ 無結果或已過期：執行檢查並快取
```

## 避免重複檢測

### 智能觸發條件

| 觸發方式 | 檢查快取？ | 說明             |
| -------- | ---------- | ---------------- |
| 手動觸發 | 否         | 強制執行完整檢查 |
| 自動觸發 | 是         | 避免重複打擾用戶 |

### 增量檢測

- 首次檢查：完整檢測所有項目
- 後續檢查：只檢查有變化的項目
- 快取結果有變化才更新

## 報告生成優化

### 只報告問題

```
[優化前] 報告所有檢查結果
[優化後] 只報告有問題的項目 + 修復建議
```

### 修復後更新快取

```
修復完成後：
→ 更新相關快取
→ 清除過期快取
→ 記錄修復日誌
```
