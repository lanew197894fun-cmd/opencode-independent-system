# 自動載入監測紀錄

## 監測目標

觀察自動載入是否影響模型學習與進化

## 觀察指標

### 1. 載入頻率

- `log.info("loaded modules for session")` 次數
- 每 session 平均載入數

### 2. 資源使用

- freeMemoryMB 變化
- loadAverage 變化
- 觸發資源限制的次數

### 3. 模型行為

- 主動探索新工具的頻率
- 依赖预加载技能的跡象

---

## 2026-03-25

### 初始狀態

- 系統：已完成 toast 改動
- 預期：使用者發送訊息後立即看到 toast 通知
- 風險評估：低

### 設計說明

#### skill-autoload.ts (首次訊息)

- 觸發條件：`isFirstMessage` 為 true（僅一次）
- 載入技能：`user-rules`（核心）、`ai-debug`（依資源）
- 目的：提供基本上下文

#### intent-router.ts (對話中)

- 觸發條件：偵測關鍵字（記憶、debug、swarm 等）
- 載入方式：按需、非一次全部
- 目的：對話驅動、按需求載入

#### 設計緩解

1. 只載入被偵測到的 intent 相關技能
2. 資源限制（低記憶/high load 時限制載入）
3. session 隔離（不同 session 獨立狀態）
4. 保留手動叫用彈性

## 測試結果應對

### ✅ 正面結果（繼續保持）

**徵兆**：

- Toast 正確顯示
- 無當機/錯誤
- 模型仍主動探索新工具
- 使用者滿意

**應對**：

1. 維持現狀
2. 持續監測記錄
3. 考慮推廣到其他功能

---

### ❌ 負面結果 A：模型依賴

**徵兆**：

- 模型變得被動
- 總是依賴預載入技能
- 很少主動建議新工具

**原因**：

- 預載入太積極
- 系統提示暗示「已經準備好」

**應對方案**：

1. 減少預載入數量（只保留 user-rules）
2. 修改 system prompt，降低「已啟用」語氣
3. 增加手動觸發比重
4. 定期重置 session 狀態

---

### ❌ 負面結果 B：效能問題

**徵兆**：

- 記憶體使用飆升
- 載入變慢
- 觸發資源限制頻繁

**原因**：

- 載入太多模組
- 未正確釋放

**應對方案**：

1. 調降 resourcePolicy 閾值
2. 增加 unload 機制
3. 限制單次最大載入數

---

### ❌ 負面結果 C：錯誤/當機

**徵兆**：

- Toast 不顯示
- 程式錯誤
- 功能異常

**原因**：

- Bus.publish 失敗
- TuiEvent 定義變更
- 匯入問題

**應對方案**：

1. 檢查日誌 error
2. 回滾程式碼
3. 加入 try-catch 保護

---

### ⚠️ 不穩定（時好時壞）

**徵兆**：

- 有時正常，有時異常
- 不同 session 表現不同

**原因**：

- session 狀態殘留
- 條件競爭
- 資源波動

**應對方案**：

1. 增加除錯日誌
2. 檢查 session 隔離
3. 統一重置時機

---

## 決策框架

### 第一步：收集證據

| 優先順序 | 檢查項目                                           |
| -------- | -------------------------------------------------- |
| 1        | 查看日誌：`log.info("loaded modules for session")` |
| 2        | 觀察 Toast 是否顯示                                |
| 3        | 檢查資源：freeMemoryMB、loadAverage                |
| 4        | 詢問使用者感受                                     |

### 第二步：分類問題

```
問題
├── 功能異常 → 當機/錯誤類
│   └── 優先：緊急修復
├── 效能下降 → 資源問題
│   └── 優先：高
└── 行為變化 → 模型依賴
    └── 優先：中（需觀察）
```

### 第三步：選擇應對

| 問題類型     | 快速判斷          | 立即動作        |
| ------------ | ----------------- | --------------- |
| Toast 不顯示 | 檢查 Bus/TuiEvent | 回滾或try-catch |
| 當機         | 查看 error 日誌   | 修復或回滾      |
| 變慢         | 檢查資源使用      | 調降閾值        |
| 模型被動     | 觀察3天對話       | 減少預載入      |

### 第四步：驗證修復

1. 小範圍測試（單一 session）
2. 觀察 24 小時
3. 如果好轉 → 全面實施
4. 如果惡化 → 回滾

---

## 快速決斷表

```
Toast 不顯示 → 回滾程式碼
當機 → 查看日誌錯誤
變慢 → 調低 resourcePolicy
模型被動 → 減少預載入 user-rules
以上皆非 → 繼續觀察
```
